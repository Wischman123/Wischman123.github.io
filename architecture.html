Ports & Adapters · Subject-free core

The assessment engine, drawn as a hexagon

One subject-free core. Everything else — the teacher UI, the LLM, the document renderer, physics itself — plugs in through a port. Dependencies only ever point inward, and a build gate proves the rule is kept rather than merely intended.

  • Driving adapters (product)
  • L2 engine core
  • L1 pure mechanism
  • Driven adapters (ports)
  • Subject plugins (L3/L4)

How a test flows through it

  1. In from the left.

    A teacher clicks in the UI, which calls the thin API over HTTP only, which calls the engine. Each layer talks inward to the next — the UI never reaches around the API to the engine.

  2. The subject-free core does the work.

    The engine assembles a Quiz from Item and Choice objects (L2) using pure drawing and equation mechanism (L1). Nothing in here knows the word “physics.”

  3. Out through ports.

    Need an LLM? Go out through the Gateway port — swap Anthropic for another backend without touching the engine. Rendering, scanning, and the question bank are the same story: adapters behind ports.

  4. Subjects plug in upward.

    At startup the engine importlib-loads the physics and DE plugins; each calls register_check() to hand the core its rules. Physics reaches into the engine — the engine never imports physics.

  5. The gate makes it true.

    import-linter scans the real import graph on every build. A stray engine → physics or engine → UI import fails CI — the rule is enforced, not hoped for.
